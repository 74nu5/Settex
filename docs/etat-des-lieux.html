01Synthèse exécutive

Settex est un projet mono-mainteneur (35 commits sur 36 d'un seul auteur) au périmètre remarquablement ambitieux pour son stade : non seulement un compilateur de langage, mais aussi un CLI, une tâche MSBuild, un serveur LSP complet et deux extensions IDE.

La fondation est saine : architecture en pipeline clairement séparée, code C# moderne (nullable activé, TreatWarningsAsErrors, central package management), build sans avertissement, et une suite de 196 tests entièrement au vert. J'ai validé le CLI de bout en bout : la compilation du sample V2 complet génère correctement les 5 fichiers appsettings*.json attendus, y compris expressions, boucles for, interpolation et overlays d'environnement.

La fragilité est à la périphérie : les fonctionnalités V2 les plus avancées (includes contenant un bloc settings, conditionnelles dans le bloc de base) et les promesses du README ne se vérifient pas toujours dans le code ; le serveur de langage embarque du code de debug et une journalisation verbeuse en production ; et les deux extensions IDE référencent des chemins de développement en dur qui les rendent non installables telles quelles.

✅ Forces

  • Pipeline de compilation propre et modulaire (Lexer → Parser → Include → Evaluator → Merger → Writer).
  • Couverture de tests unitaires solide sur le cœur (181 tests).
  • Diagnostics précis ligne/colonne au format MSBuild, cliquables dans l'IDE.
  • Écriture conditionnelle des fichiers (n'écrit que si le contenu change).
  • Fusion profonde (deep-merge) correcte et bien testée.
  • CLI soigné (Spectre.Console) fonctionnel de bout en bout.

⚠️ Faiblesses

  • README fortement désynchronisé de la syntaxe réellement supportée.
  • Includes + settings cassés end-to-end et non testés.
  • Bug d'exécution : l'extension VS appelle une commande CLI inexistante.
  • Extensions non déployables (chemins bin/Debug/net10.0 en dur).
  • LSP : code de debug résiduel, logs verbeux, document non thread-safe.
  • Couverture d'intégration mince à travers le compilateur complet.

04État technique & qualité

Build & conventions

  • Nullable activé, ImplicitUsings, LangVersion=latest.
  • TreatWarningsAsErrors=true — discipline forte.
  • Central Package Management (Directory.Packages.props).
  • Build Core en Release : 0 avertissement, 0 erreur.
  • Versions de packages figées pour éviter CVE-2025-55247 (MSBuild).

Tests

  • 196 tests (TUnit) — 181 core + 15 LSP, 100 % au vert.
  • Bien couvert : expressions, parser, lexer, evaluator, merger, writer.
  • Faible : intégration end-to-end via SettexCompiler (includes, let, for non testés bout-en-bout).
  • Pas de tests unitaires sur la tâche MSBuild ni le CLI.

Concurrence & robustesse (extension VS)

L'historique git montre un travail itératif intense sur les race conditions et le disposal (5+ commits dédiés). Le résultat est nettement plus robuste qu'à ses débuts : débounce par fichier via ConcurrentDictionary<string,CancellationTokenSource>, capture des options sur le thread UI, nettoyage atomique avec TryRemove + ReferenceEquals. Il subsiste toutefois deux fragilités réelles :

  • Le disposal du CancellationTokenSource repose sur un délai magique de 100 ms dans un Task.Run fire-and-forget — heuristique, pas garantie.
  • L'annulation ne tue pas le sous-processus dotnet déjà lancé (net472 sans WaitForExitAsync) : fuite de process possible, et aucun timeout de compilation.

06Bugs & dette prioritaires

✅ Corrigé — Le build auto de l'extension VS invoquait une commande CLI inexistante
SettexBuildService invoquait dotnet "Settex.Cli.dll" compile "<fichier>", mais le CLI n'expose que la commande build — la compilation manuelle et le compile-on-save échouaient systématiquement à l'exécution. Corrigé (compile → build) et validé end-to-end : l'invocation génère bien les fichiers appsettings*.json dans le répertoire du .settex.
extensions/vs-settex/Settex.VisualStudio/SettexBuildService.cs:70 · src/Settex.Cli/Program.cs:15
✅ Corrigé — Le VSIX de l'extension VS ne compilait pas
Le projet épinglait un SDK MSBuild inexistant (CodingWithCalvin.VsixSdk 1.0.0, max publié = 0.4.0) et des versions de packages non restaurables (VS SDK 17.12.40459, StreamJsonRpc 2.19.27), le tout transformé en erreurs par TreatWarningsAsErrors. Corrigé : SDK→0.4.0, VS SDK→17.13.x, StreamJsonRpc→2.21.10, adaptation de ILanguageClient à la surface 17.13, RunningDocumentTable.Unadvise au lieu de Dispose (supprimé), et packaging des grammaires/snippets/pkgdef/licence en Content. Un VSIX complet est produit avec MSBuild 18.9. Contrepartie : la version minimale de Visual Studio passe à 17.13.
extensions/vs-settex/Settex.VisualStudio/*.cs · Directory.Packages.props · source.extension.vsixmanifest
✅ Corrigé — Binaires embarqués dans les deux extensions
Les deux extensions embarquent désormais leurs binaires. Côté Visual Studio : le serveur LSP et le CLI (publiés framework-dependent net10.0) sont embarqués dans le VSIX sous LanguageServer/ et Cli/ via une cible MSBuild, résolus relativement au dossier de l'assembly de l'extension (et non AppDomain.BaseDirectory, qui pointe vers devenv). Côté VS Code : le serveur est publié dans server/ via un script bundle-server branché sur vscode:prepublish, et résolu depuis context.asAbsolutePath avec repli sur l'arbre source. VSIX ≈ 2,9 Mo (VS) et 2,19 Mo (VS Code), self-contained (le runtime .NET 10 doit être présent). Vérifié : CLI et serveur extraits des deux paquets se lancent et fonctionnent via dotnet.
vs-settex : Settex.VisualStudio.csproj (PublishSettexRuntimeBinaries) · vscode-settex : package.json (bundle-server) · extension.ts
🟠 Majeur — Includes contenant settings
Fonctionnalité annoncée mais cassée end-to-end et non couverte par un test d'intégration ; un test d'IncludeResolver affirme même produire plusieurs SettingsBlockNode, configuration que l'évaluateur rejette ensuite.
🟠 Majeur — Serveur LSP : code de debug en production
Bloc de debug codé en dur sur la valeur littérale "Comparison" dans la complétion, journalisation très verbeuse au niveau Information (dont la sérialisation JSON complète du modèle à chaque survol), et SetMinimumLevel(LogLevel.Debug) — bruit et coût significatifs.
src/Settex.LanguageServer/SettexCompletionHandler.cs · HoverOverlayFormatter.cs · Program.cs:18
🟠 Majeur — Document LSP non thread-safe
SettexDocument est muté par Reparse() (passe ast à null puis réassigne) pendant que complétion/hover lisent l'AST en concurrence, sans synchronisation ni versioning. Le « sync incrémental » annoncé est en réalité un full-reparse à chaque frappe, avec relecture disque des includes.

Dette technique diffuse

  • Code mort dans le LSP (méthodes jamais appelées, variables inutilisées, logique dupliquée entre ScopeResolver et SettexReferencesHandler).
  • Heuristiques de portée codées en dur (line ≤ start + 1000) faute de position de fin dans SourceLocation.
  • LocationToRange suppose que tout tient sur une seule ligne → ranges faux pour les nœuds multi-lignes.
  • Conversions numériques double ↔ decimal ↔ long potentiellement lossy.
  • Pas d'écriture atomique des fichiers de sortie (pas de temp + rename).
  • catch {} vides masquant des erreurs (disposal, PATH).
  • Commande de démo helloWorld laissée dans l'extension VS Code.

07Évaluation critique

« Un excellent projet d'auteur au potentiel réel, freiné par un écart entre l'ambition affichée et l'état de finition. »

Sur le plan de l'ingénierie, Settex est au-dessus de la moyenne des projets à ce stade : le découpage en phases est textbook, le code est idiomatique et discipliné (warnings-as-errors), et l'existence d'un serveur LSP maison avec hover contextuel affichant les valeurs résolues par environnement est une fonctionnalité différenciante et non triviale. Le fait que les 196 tests passent et que le CLI produise un JSON correct sur un sample V2 riche prouve que le concept fonctionne réellement.

Sur le plan du produit, le projet souffre du syndrome classique du « 80 % fait, 80 % restant ». L'ambition (langage + CLI + MSBuild + LSP + 2 extensions) a été déployée en largeur avant d'être consolidée en profondeur. Les conséquences concrètes :

  • Confiance érodée : un utilisateur qui suit le README échoue dès le premier env. La documentation, censée vendre le produit, devient un passif.
  • Fonctionnalités « fantômes » : includes+settings, conditionnelles de base, guillemets simples, objets anonymes sont annoncés mais absents ou cassés — le périmètre réel est plus petit que le périmètre communiqué.
  • Livrables non livrables : les deux extensions, vitrine de l'expérience développeur, ne s'installent pas hors arbre de développement, et le build auto de l'extension VS est inopérant (mauvaise commande CLI).

La question de la pertinence même mérite d'être posée : Settex entre en concurrence avec des approches établies (transformations de config .NET, variables d'environnement, JSON + dotnet user-secrets, ou des langages comme CUE/Dhall/Jsonnet). Sa proposition de valeur — un DSL lisible avec overlays et deep-merge — est réelle mais étroite ; elle ne se justifiera que si l'expérience « ça marche du premier coup » est irréprochable. Aujourd'hui, ce n'est pas le cas.

Ce qui impressionne

  • LSP maison fonctionnel
  • Discipline de build
  • Deep-merge correct
  • Diagnostics IDE précis

Ce qui inquiète

  • Doc ≠ code
  • Intégration peu testée
  • Fonctions V2 fragiles
  • Debug en prod (LSP)

Ce qui bloque

  • Extensions non déployables
  • Build VS inopérant
  • Includes+settings cassés
  • Placeholders partout

Conclusion
Settex n'est pas un projet en difficulté — c'est un projet en avance sur sa consolidation. Les fondations techniques sont bonnes et le concept est prouvé. Un cycle de « resserrage » ciblé (aligner doc et code, rendre les livrables installables, corriger les 3-4 bugs bloquants, ajouter des tests d'intégration) le ferait passer d'un prototype impressionnant à un outil réellement adoptable. La priorité n'est pas d'ajouter des fonctionnalités, mais de rendre vraies celles déjà annoncées.
