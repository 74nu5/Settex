Pourquoi la note remonte (5,8 → 7,2)
La consolidation est réelle et vérifiée : tous les bugs de langage de la 2e revue sont corrigés et couverts par des tests, l'extension VS Code s'installe et s'active, le client LSP VS ne laisse plus de process orphelin, la navigation cross-fichier et l'outline sont justes, la sortie est passée en deltas alignés sur le layering .NET, et un lint de couverture répond au besoin d'origine. 262 tests (contre 228), 0 avertissement. Pourquoi pas plus haut
Les deux plus gros trous ont été fermés dans la foulée de cette revue (diagnostics sémantiques & couverture en éditeur ; conflit de type localisé — voir §03). Il reste des défauts réels : le piège d'indexation des tableaux de .NET casse silencieusement la sémantique « les tableaux se remplacent » (les deux modes), le mode delta ne valide pas les types base ↔ overlay, et la couche éditeur garde des ranges mono-ligne et une étendue de scope imparfaite. Plusieurs de ces défauts sont des angles morts des correctifs récents — c'est noté explicitement.
Verdict en une phrase
Un compilateur d'auteur désormais solide et bien testé, avec un écosystème qui fonctionne — mais dont la couche éditeur et quelques pièges de layering runtime le maintiennent en bêta soignée plutôt qu'en 1.0.

01Contexte & méthode

Entre-temps, 11 PR ont été mergées, fermant l'intégralité des recommandations de la revue précédente : packaging VS Code, lexer des négatifs, block-scoping, client LSP VS, dette LSP (snapshot, portée, URI, cross-fichier, outline), interpolation, overflow décimal, includes en losange, détection du runtime .NET 10, projet de tests CLI, polish docs, et enfin le modèle de sortie (deltas + lint de couverture).

  • Deux relecteurs indépendants à yeux neufs : un sur le cœur compilateur et le code neuf (delta/couverture), un sur le serveur de langage. Chaque défaut ci-dessous est marqué reproduit ou raisonné.
  • Regard critique assumé : quand un correctif de cette session a laissé ou créé un angle mort, c'est signalé issu du dernier cycle — un correctif qui déplace le défaut d'un cran mérite d'être dit.

03Défauts trouvés à froid

🔴 Élevés

✅ Corrigé — le serveur de langage montre désormais erreurs sémantiques & dérive de couverture
SettexDocument.Parse s'arrêtait après la résolution des includes, sans lancer l'Evaluator ni le CoverageAnalyzer : variable indéfinie, conflit de type, et surtout la dérive inter-environnements n'apparaissaient qu'au CLI/build, jamais dans l'éditeur. Corrigé : une phase d'évaluation + couverture est ajoutée à Parse (ignorée si une phase antérieure a déjà échoué), et ses diagnostics — erreurs sémantiques (STX401) et warnings de couverture (STX501) — sont publiés dans l'éditeur. Vérifié : variable indéfinie → erreur, dérive → warning nommant la clé, conflit de type → erreur ; 4 tests de non-régression.
src/Settex.LanguageServer/SettexDocument.cs (Parse, ToLspDiagnostic) · reste : la couverture n'a pas de localisation source → ancrée en tête de fichier
La sémantique « les tableaux se remplacent » est fausse à l'exécution reproduit
Base Hosts = ["a","b","c"], env Hosts = ["x"]. Settex présente le remplacement complet (le Merger l'applique). Mais .NET aplatit les tableaux en clés indexées (Hosts:0/1/2) et superpose par index les deux fichiers → valeur effective ["x","b","c"], pas ["x"]. Vrai en mode delta ET --merged (le fichier de base contient toujours le tableau). Aucun avertissement, aucune doc du piège. Le format ne peut pas exprimer « raccourcir un tableau » à travers les couches.
src/Settex.Core/Writing/JsonWriter.cs · Merger.cs — au minimum : documenter le piège ; idéalement, avertir quand un env raccourcit un tableau présent en base
✅ Corrigé — le conflit de type est désormais un diagnostic localisé
MergerException n'héritant pas d'EvaluatorException, sur settings { Foo = 1 } settings { Foo { Bar = 2 } } l'erreur tombait dans le catch-all → « Unexpected error… » sans ligne/colonne. Corrigé : l'Evaluator enveloppe ses appels de merge (SafeMerge) et convertit la MergerException en EvaluatorException avec la localisation du bloc fautif ; le compilateur a aussi un filet de sécurité en phase 3. Vérifié : erreur à (2,1), plus « Type mismatch for key 'Foo' » ; test de non-régression ajouté.
src/Settex.Core/Evaluation/Evaluator.cs (SafeMerge) · SettexCompiler.cs

🟠 Moyens

Le mode delta n'effectue aucune validation de type base ↔ overlay reproduit issu du dernier cycle
Base Foo = 1, env Foo { Bar = 2 } : en mode delta (défaut) ça compile et écrit un JSON incohérent (base Foo:1 + env Foo:Bar:2), qui casse le binding .NET. Le même fichier est rejeté en --merged. La validité dépend donc d'un drapeau de sortie, et c'est le défaut par défaut qui masque le problème — parce que les overlays ne sont jamais fusionnés contre la base en mode delta.
src/Settex.Core/Evaluation/Evaluator.cs · JsonWriter.cs — valider la compatibilité base/overlay indépendamment du mode
Ranges de symboles mono-ligne → containment LSP violée raisonné
LocationToRange reste mono-ligne (SourceLocation sans EndLine/EndColumn). DocumentSymbol l'utilise pour le Range des blocs settings/env/imbriqués : un bloc lignes 2-40 a un range = ligne 2, alors que ses enfants sont lignes 3-39, donc hors du parent. Le LSP exige l'inclusion des ranges enfants → breadcrumbs / sticky-scroll / outline cassés ou mal imbriqués selon le client.
src/Settex.LanguageServer/SettexDocument.cs (LocationToRange) — nécessite EndLine/EndColumn dans SourceLocation (change Core)
L'étendue de scope sous-couvre les lignes de fin / accolade raisonné issu du dernier cycle
Le nouveau calcul d'étendue ne pousse EndLine que par la ligne de début des nœuds traversés. Toute position entre la dernière ligne de contenu et le } fermant (lignes vides, accolade) retombe sur le scope parent ; un env { settings {} } vide ne couvre que sa ligne de déclaration. Impact borné (les identifiants sont sur des lignes couvertes, donc def/hover marchent en général) — mais c'est le défaut miroir de l'ancienne heuristique +1000.
src/Settex.LanguageServer/ScopeInfo.cs · ScopeResolver.cs — étendre jusqu'à la ligne de l'accolade fermante
Go-to-definition in-file sur doc enregistré : URI reconstruite, casse possiblement altérée raisonné issu du dernier cycle
ToLspLocation reconstruit l'URI depuis le chemin FS (FromFileSystemPath) dès que FilePath est non nul — donc même pour un saut dans le fichier courant enregistré, l'URI fait un aller-retour qui peut changer la casse de la lettre de lecteur sur Windows. Le client (VS Code envoie souvent une lettre minuscule) peut alors ouvrir un nouvel onglet au lieu de révéler. Les docs non enregistrés (FilePath == null) retombent correctement sur l'URI de la requête.
src/Settex.LanguageServer/SettexDocument.cs (ToLspLocation) — réutiliser l'URI de la requête pour le fichier courant
Aucun moyen d'émettre un ${…} littéral — $ n'a pas d'échappement reproduit
L'interpolation se déclenche sur un simple str.Contains("${"), et l'échappement du lexer se limite à \" \\ \n \r \t (\$ lève « Invalid escape sequence »). Toute chaîne contenant légitimement ${…} (templates shell/docker-compose, regex, jetons .NET) est forcée en interpolation : "${HOME}" échoue « Variable 'HOME' is not defined » sans échappatoire évidente.
src/Settex.Core/Lexer/Lexer.cs · Parser.cs — supporter \$ ou $${

🟡 Faibles

  • Couverture — clé présente dans tous les env mais pas la base : jamais signalée reproduit. Défendable comme choix, mais c'est précisément la dérive qui mord quand on ajoute un nouvel environnement. À trancher explicitement.
  • Couverture — conflit objet/feuille : Shape = 1 (Dev) vs Shape { Inner = 2 } (Prod) produit deux avertissements contradictoires (« Shape absent de Prod », « Shape.Inner absent de Dev ») sans jamais nommer le vrai problème (un conflit de type) reproduit.
  • Scanner d'accolades d'interpolation ignore les chaînes internes : "${ y == \"}\" }" s'arrête sur le } dans la chaîne interne raisonné.
  • Dédup/cycle d'includes comparent des chemins GetFullPath : liens symboliques/durs et noms courts 8.3 peuvent les contourner (borné par MaxIncludeDepth, donc pas de blocage) raisonné.
  • LSP : includes lus du disque, ignorant les buffers non sauvegardés ; pas de re-parse quand un fichier inclus change raisonné.
  • LSP : itérateurs de for non modélisés comme variables de scope → go-to-def sur l'itérateur saute vers une variable homonyme externe raisonné.
  • LSP : handlers sans try/catch ; LocationToRange calcule Column-1 → une colonne 0 produirait une position négative raisonné.
  • Highlighter des docs ignore les commentaires # (Settex en supporte deux styles) — latent, aucun exemple actuel ne l'utilise reproduit.

05Verdict & priorités

Le projet a franchi un vrai palier depuis la 2e revue : le cœur du langage est correct et couvert, l'écosystème d'extensions fonctionne, et le modèle de sortie répond enfin au problème d'origine (la dérive de configuration). La note remonte à 7,2 parce que ces gains sont réels et vérifiés, pas décrétés.

Il reste en bêta soignée pour trois raisons honnêtes : (1) la couche éditeur ne montre pas les diagnostics qui comptent — évaluation et couverture — ce qui vide en partie la promesse de l'IDE ; (2) le piège d'indexation des tableaux de .NET fait mentir une sémantique documentée, dans les deux modes ; (3) quelques trous de robustesse (conflit de type non localisé, validation delta absente) qui, ironiquement, incluent des angles morts des correctifs de ce cycle.
