Pourquoi la note remonte (5,8 → 8,0)
La consolidation est réelle et vérifiée : tous les bugs de langage de la 2e revue sont corrigés et couverts par des tests, l'extension VS Code s'installe et s'active, le client LSP VS ne laisse plus de process orphelin, la navigation cross-fichier et l'outline sont justes, la sortie est passée en deltas alignés sur le layering .NET, et un lint de couverture répond au besoin d'origine. Pourquoi pas plus haut
Tous les défauts élevés et moyens relevés par cette revue ont depuis été traités (voir §03) — y compris ceux qui étaient des angles morts des correctifs du cycle précédent, signalés comme tels. Ce qui subsiste relève de l'edge case ou de l'externe : le scanner d'accolades d'interpolation ignore les chaînes internes, les liens symboliques peuvent contourner la dédup d'includes, et les extensions ne sont pas publiées sur les marketplaces. Correction d'une affirmation antérieure de ce rapport : le runtime .NET 10 est publié (LTS), pas une préversion — la dépendance runtime des utilisateurs est donc normale, et VS Code l'acquiert même automatiquement. L'hygiène de build est désormais réglée : global.json épingle un SDK GA (allowPrerelease: false), vérifié 0 avertissement et 305 tests au vert sous ce SDK.
Verdict en une phrase
Un compilateur d'auteur désormais solide et bien testé, avec un écosystème qui fonctionne — mais dont la couche éditeur et quelques pièges de layering runtime le maintiennent en bêta soignée plutôt qu'en 1.0.

01Contexte & méthode

Cette revue est la troisième du projet, et la seconde menée à froid. Entre-temps, 11 PR ont été mergées, fermant l'intégralité des recommandations de la revue précédente : packaging VS Code, lexer des négatifs, block-scoping, client LSP VS, dette LSP (snapshot, portée, URI, cross-fichier, outline), interpolation, overflow décimal, includes en losange, détection du runtime .NET 10, projet de tests CLI, polish docs, et enfin le modèle de sortie (deltas + lint de couverture).

  • Vérité terrain reproduite : build de la solution (0 avertissement), 262 tests (222 core + 22 LSP + 7 CLI + 4 MSBuild + 7 extension) tous au vert, et une série de compilations CLI adverses sur les cas limites des nouveautés (deltas, --merged, couverture, tableaux, conflits de type).
  • Deux relecteurs indépendants à yeux neufs : un sur le cœur compilateur et le code neuf (delta/couverture), un sur le serveur de langage. Chaque défaut ci-dessous est marqué reproduit ou raisonné.
  • Regard critique assumé : quand un correctif de cette session a laissé ou créé un angle mort, c'est signalé issu du dernier cycle — un correctif qui déplace le défaut d'un cran mérite d'être dit.

02Ce qui tient (vérifié)

Contrairement à la 2e revue, ces points ne sont plus des promesses de commentaires — ils ont été rejoués :

Langage & compilateur

  • Tous les bugs de langage de la 2e revue corrigés et testés (soustraction, portée des let, interpolation à jetons stricts, overflow décimal localisé).
  • Includes en losange dédupliqués (include-once), casse selon le système de fichiers, profondeur bornée.

  • Navigation cross-fichier et outline corrects (chemins propagés, symboles inclus filtrés).
  • Extensions : VSIX VS Code fonctionnel, process LSP VS géré, runtime .NET 10 détecté avec message actionnable.
  • Couverture de tests réelle : 262 tests, assertions d'exception typées, projet CLI black-box, CoverageAnalyzer unitairement testé (déterministe, sans faux positif sur les clés de base).

03Défauts trouvés à froid

🔴 Élevés

✅ Corrigé — le serveur de langage montre désormais erreurs sémantiques & dérive de couverture
SettexDocument.Parse s'arrêtait après la résolution des includes, sans lancer l'Evaluator ni le CoverageAnalyzer : variable indéfinie, conflit de type, et surtout la dérive inter-environnements n'apparaissaient qu'au CLI/build, jamais dans l'éditeur. Corrigé : une phase d'évaluation + couverture est ajoutée à Parse (ignorée si une phase antérieure a déjà échoué), et ses diagnostics — erreurs sémantiques (STX401) et warnings de couverture (STX501) — sont publiés dans l'éditeur. Vérifié : variable indéfinie → erreur, dérive → warning nommant la clé, conflit de type → erreur ; 4 tests de non-régression.
src/Settex.LanguageServer/SettexDocument.cs (Parse, ToLspDiagnostic) · reste : la couverture n'a pas de localisation source → ancrée en tête de fichier
✅ Adressé — le piège d'indexation des tableaux est détecté, averti & documenté
.NET superpose les tableaux par index (base ["a","b","c"] + env ["x"] → ["x","b","c"] à l'exécution, vérifié empiriquement via le binder .NET réel). Settex ne peut pas changer ce comportement du provider — aucun « clear array » n'existe dans les fichiers appsettings, et un padding par null ne ferait que produire ["x",null,null]. Le correctif mûr : un ArrayLayeringAnalyzer avertit à la compilation quand un environnement raccourcit un tableau présent en base (le cas où la fuite est garantie), et le piège est désormais documenté (README + guide du langage). Vérifié : Production 3→1 avertit, Staging 3→3 non ; 5 tests unitaires + tests compilateur/LSP. Reste une limite .NET intrinsèque, pas un bug Settex.
src/Settex.Core/Compilation/ArrayLayeringAnalyzer.cs (nouveau) — surfacé au CLI, au build MSBuild et dans l'éditeur
✅ Corrigé — le conflit de type est désormais un diagnostic localisé
MergerException n'héritant pas d'EvaluatorException, sur settings { Foo = 1 } settings { Foo { Bar = 2 } } l'erreur tombait dans le catch-all → « Unexpected error… » sans ligne/colonne. Corrigé : l'Evaluator enveloppe ses appels de merge (SafeMerge) et convertit la MergerException en EvaluatorException avec la localisation du bloc fautif ; le compilateur a aussi un filet de sécurité en phase 3. Vérifié : erreur à (2,1), plus « Type mismatch for key 'Foo' » ; test de non-régression ajouté.
src/Settex.Core/Evaluation/Evaluator.cs (SafeMerge) · SettexCompiler.cs

🟠 Moyens

✅ Corrigé — la compatibilité de type base ↔ overlay est validée dans les deux modes
Base Foo = 1, env Foo { Bar = 2 } : en mode delta (défaut) ça compilait et écrivait un JSON incohérent, alors que --merged le rejetait — la validité dépendait d'un drapeau de sortie, et le défaut masquait le problème. Corrigé : l'Evaluator valide chaque overlay contre la base après évaluation, indépendamment du mode de sortie, via Merger.AreCompatible — la règle est donc littéralement celle qu'applique la fusion, sans risque de dérive. Le diagnostic nomme le chemin complet et l'environnement (Type mismatch for key 'Server.Port' in environment 'Dev'), mieux que le message historique limité à la clé feuille. Vérifié : delta et merged rejettent au même endroit ; overlays compatibles (objets, tableaux, primitives, clés nouvelles) inchangés ; 3 tests de non-régression.
src/Settex.Core/Evaluation/Evaluator.cs (ValidateOverlayAgainstBase) · Merging/Merger.cs (AreCompatible, DescribeNodeType)
✅ Corrigé — ranges de symboles multi-lignes, containment LSP respectée
LocationToRange était mono-ligne (SourceLocation sans EndLine/EndColumn) : un bloc lignes 2-40 avait un range = ligne 2, ses enfants lignes 3-39 tombaient hors du parent → breadcrumbs / outline cassés. Corrigé : SourceLocation porte désormais EndLine/EndColumn (nullables, avec accesseurs effectifs rétrocompatibles), le parser spanne tous les nœuds à blocs jusqu'à leur accolade/crochet fermant (settings, env, bloc imbriqué, for, objet taggé, tableau, fichier), et LocationToRange produit un range multi-ligne. Les positions sont aussi bornées à zéro (ferme au passage le risque de coordonnée négative).
src/Settex.Core/Diagnostics/SourceLocation.cs · Parser.cs (SpanTo) · SettexDocument.cs (LocationToRange)
✅ Corrigé — l'étendue de scope couvre tout le corps, jusqu'à l'accolade fermante
L'étendue n'était poussée que par la ligne de début des nœuds traversés : toute position entre la dernière ligne de contenu et le } fermant retombait sur le scope parent, et un env { settings {} } vide ne couvrait que sa ligne de déclaration. Corrigé par le même changement Core : ScopeInfo initialise sa fin depuis la vraie fin de bloc de l'AST. Vérifié : ligne vide et accolade fermante d'un env résolvent bien vers l'env, un corps d'env vide aussi, et le contenu après l'env retombe correctement sur le global.
src/Settex.LanguageServer/ScopeInfo.cs (EffectiveEndLine) — le défaut miroir de l'ancienne heuristique +1000 est fermé
✅ Corrigé — le go-to-definition in-file réutilise l'URI de la requête
ToLspLocation reconstruisait l'URI depuis le chemin FS dès que FilePath était non nul : même pour un saut dans le fichier courant, l'aller-retour pouvait changer la casse de la lettre de lecteur sur Windows, et un client comparant les URI ouvrait un nouvel onglet au lieu de révéler la position. Corrigé : quand la cible est le document courant (comparaison de chemins selon la casse du système de fichiers), l'URI de la requête est réutilisée telle quelle ; sinon seulement, elle est construite depuis le chemin. Vérifié : chemin de casse différente → URI de requête conservée ; autre fichier → URI de ce fichier.
src/Settex.LanguageServer/SettexDocument.cs (ResolveTargetUri)
✅ Corrigé — $${ échappe un ${…} littéral
Toute chaîne contenant légitimement ${…} (templates shell/compose, regex, jetons destinés à un autre outil) était forcée en interpolation, sans échappatoire (\$ levait « Invalid escape sequence »). Corrigé : $${ produit un ${ littéral. L'échappement est traité là où la décision d'interpolation se prend (le parser) et non dans le lexer — un \$ décodé en $ aurait été indiscernable et se serait heurté à l'ambiguïté avec "\\$". Le message d'erreur de \$ oriente désormais vers la bonne syntaxe. Vérifié : littéral préservé, variable inexistante non évaluée, et échappement + interpolation réelle dans la même chaîne ; 3 tests, syntaxe documentée (README, guide, aide-mémoire).
src/Settex.Core/Parser/Parser.cs (ParseInterpolatedString) · Lexer.cs (message d'aide)

🟡 Faibles

  • ✅ Tranché — Couverture : clé présente dans tous les env mais pas la base → volontairement non signalée. La config est correcte telle quelle, et le « correctif » évident (inventer une valeur de base) est souvent pire pour ce qui doit être décidé par environnement — une valeur plausible mais fausse s'appliquerait silencieusement. Surtout, la crainte du « on ajoute un env plus tard et la clé y manque » est infondée : vérifié qu'ajouter un environnement sans la clé déclenche immédiatement l'avertissement existant — le filet se referme au moment exact où le risque devient réel. Décision documentée dans le code et pour l'utilisateur, verrouillée par un test de non-régression.
  • Couverture — conflit objet/feuille : Shape = 1 (Dev) vs Shape { Inner = 2 } (Prod) produit deux avertissements contradictoires (« Shape absent de Prod », « Shape.Inner absent de Dev ») sans jamais nommer le vrai problème (un conflit de type) reproduit.
  • Scanner d'accolades d'interpolation ignore les chaînes internes : "${ y == \"}\" }" s'arrête sur le } dans la chaîne interne raisonné.
  • Dédup/cycle d'includes comparent des chemins GetFullPath : liens symboliques/durs et noms courts 8.3 peuvent les contourner (borné par MaxIncludeDepth, donc pas de blocage) raisonné.
  • ✅ Corrigé — LSP : includes lus du disque, ignorant les buffers non sauvegardés, sans re-parse quand un fichier inclus changeait. L'IncludeResolver accepte désormais un fournisseur de contenu (les buffers ouverts gagnent sur le disque) et expose les fichiers résolus ; le workspace suit ces dépendances et ré-analyse les documents dépendants à l'ouverture, la modification et la fermeture d'un fichier inclus. Vérifié : buffer non sauvegardé pris en compte, dépendants rafraîchis, retour au disque à la fermeture, documents non concernés épargnés.
  • ✅ Corrigé — LSP : itérateurs de for non modélisés comme variables de scope (go-to-def sautait vers une variable homonyme externe). L'itérateur est désormais enregistré comme un LetNode synthétique pointant sur la boucle, donc il masque correctement les homonymes externes ; le hover l'affiche explicitement comme itérateur avec la collection parcourue, plutôt que de laisser croire que sa valeur est la collection.
  • ✅ Corrigé — LSP : handlers sans try/catch (un helper qui lève faisait échouer la requête au lieu de dégrader). Les cinq handlers (hover, definition, references, symbols, complétion) enveloppent désormais leur corps : l'échec est journalisé et la requête retourne un résultat vide, tandis qu'une annulation est relayée telle quelle (ce n'est pas une erreur). Le risque de position négative (Column-1) était déjà fermé par le bornage dans LocationToRange. Vérifié par un smoke test LSP réel : le serveur démarre et annonce bien les cinq providers.
  • ✅ Corrigé — LSP : hover overlay insensible aux colonnes et aveugle aux blocs imbriqués. Il ne parcourait que les statements directs, donc survoler Port dans settings { Server { Port = … } } ne déclenchait rien ; et n'importe quelle position sur la ligne d'une assignation la déclenchait. La recherche descend désormais dans les blocs imbriqués en accumulant le chemin complet (Server.Port), et les assignations portent une étendue (chemin → fin de valeur) permettant un vrai test de position. Vérifié par des tests de handler : hover imbriqué OK, hover sur un mot d'un commentaire suivant l'assignation ne déclenche plus.
  • Highlighter des docs ignore les commentaires # (Settex en supporte deux styles) — latent, aucun exemple actuel ne l'utilise reproduit.

05Verdict & priorités

Le projet a franchi un vrai palier depuis la 2e revue : le cœur du langage est correct et couvert, l'écosystème d'extensions fonctionne, et le modèle de sortie répond enfin au problème d'origine (la dérive de configuration). La note remonte à 7,2 parce que ces gains sont réels et vérifiés, pas décrétés.

Il reste en bêta soignée pour trois raisons honnêtes : (1) la couche éditeur ne montre pas les diagnostics qui comptent — évaluation et couverture — ce qui vide en partie la promesse de l'IDE ; (2) le piège d'indexation des tableaux de .NET fait mentir une sémantique documentée, dans les deux modes ; (3) quelques trous de robustesse (conflit de type non localisé, validation delta absente) qui, ironiquement, incluent des angles morts des correctifs de ce cycle.
