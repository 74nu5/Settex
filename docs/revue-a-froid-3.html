Deux défauts élevés et un moyen sont corrigés, et le rapport est annoté en conséquence :

  • H1 — CI de build et de test : en place, sur Ubuntu et Windows, plus la compilation de l'extension VS Code. fait
  • H5 — débordement de pile du lexer : corrigé, 200 000 lignes vides passent désormais. fait
  • M6 — CI contredisant l'épinglage GA : désamorcé avant qu'il ne se déclenche. fait
  • Nouveau — validation des noms d'environnement dépendante de l'OS : trouvé par la CI elle-même, à sa toute première exécution, puis corrigé. fait

La troisième revue concluait « 0 défaut élevé restant ». Cette relecture en trouve cinq. Ce n'est pas un retour en arrière du produit : les fonctionnalités livrées entre-temps — sortie delta, lint de couverture, diagnostics sémantiques dans l'éditeur, navigation inter-fichiers — fonctionnent et répondent au besoin d'origine. Le problème est ailleurs.

Le fait marquant de cette revue

Trois des cinq défauts élevés sont des régressions introduites par la campagne de correctifs elle-même, et aucune n'a été détectée : les 305 tests sont restés verts du début à la fin. La cause commune était structurelle — il n'existait aucune intégration continue de build ou de test. Chaque correctif a été validé localement, à la main, par la personne qui l'écrivait. C'est le premier point corrigé depuis ; voir l'encadré ci-dessus.

Autrement dit : la qualité du code livré est réelle, mais le filet qui devrait la préserver n'existe pas. C'est le vrai sujet de cette revue, et c'est ce qui justifie la note en baisse malgré des fonctionnalités en hausse.

03Défauts élevés

H1 — Aucune intégration continue de build ou de test corrigé

Constat initial : .github/workflows/ ne contenait qu'un déploiement du site de documentation, filtré sur docs-site/**. Les 305 tests ne s'exécutaient sur aucun push ni aucune pull request. La quinzaine de PR de la campagne de correctifs a été fusionnée sans la moindre vérification automatique ; les régressions ci-dessous en sont la démonstration directe.

Correctif : build et tests de la solution sur Ubuntu et Windows, plus la compilation du TypeScript de l'extension VS Code — à chaque push sur main et chaque pull request. Trois choix structurants :

  • Le SDK vient de global-json-file et non d'une version inscrite dans le workflow : la CI et le poste de développement ne peuvent plus diverger.
  • Aucun filtre de chemins, puisque c'est précisément ce filtrage qui avait laissé passer M6.
  • Deux systèmes d'exploitation, parce que IncludeResolver compare les chemins différemment selon la plateforme — ce n'était pas du remplissage, voir ci-dessous.
La CI a payé à sa première exécution

Le job Ubuntu, qui n'avait jamais tourné, a échoué immédiatement sur trois tests. Deux étaient des tests non portables. Le troisième était un défaut produit que ni la revue ni les relecteurs n'avaient vu : JsonWriter validait les noms d'environnement avec Path.GetInvalidFileNameChars(), dont le contenu dépend de la plateforme — une quarantaine de caractères sur Windows, uniquement NUL et / sur Unix.

Un env "Dev<>ment" était donc refusé sur Windows et accepté sur Linux. Le nom devenant une partie du fichier appsettings.{Environment}.json, un build Linux produisait des fichiers qu'un checkout Windows ne peut même pas créer. La validation utilise désormais le jeu le plus strict partout, avec un test paramétré épinglant chaque caractère concerné.

H2 — Le LSP confond les positions entre fichiers inclus et fichier courant reproduit

ScopeResolver.cs:17 · ScopeInfo.cs:105 · SettexHoverHandler.cs:610

L'AST manipulé par le serveur est aplati : les instructions des fichiers inclus y sont fusionnées avec celles du fichier courant. Or ni IsPositionInLocation ni ScopeInfo.ContainsPosition ne comparent le chemin du fichier — ils raisonnent uniquement en ligne et colonne. Les lignes 5 à 10 d'un fichier inclus se superposent donc aux lignes 5 à 10 du fichier ouvert.

Reproduit : avec un let host = "MAIN-HOST" local et un env "Dev" { let host = "DEV-HOST" } dans le fichier inclus, survoler host annonce Scope: Env "Dev" et la valeur "DEV-HOST". Le « aller à la définition » saute dans le fichier inclus. Les deux réponses sont fausses. Le survol d'une affectation est touché de la même façon, et SettexReferencesHandler hérite du défaut.

Aggravé par un correctif de la campagne : donner aux scopes leur étendue réelle issue de l'AST rend la superposition plus fiable, donc plus nuisible. À noter que SettexDocumentSymbolHandler fait déjà le bon test via IsFromSameFile — l'asymétrie confirme l'oubli.

H3 — Tout fragment d'include affiche une erreur permanente dans l'éditeur régression reproduit

SettexDocument.cs:161 → Evaluator.cs:185

Le serveur exécute désormais l'évaluateur sur chaque tampon ouvert, et ValidateStructure exige au moins un bloc settings. Un fragment destiné à être inclus n'en a pas, par construction.

Le dépôt livre exactement un tel fichier : samples/common.settex déclenche STX401 — File must contain at least one 'settings' block. Le README présente ce motif comme la façon normale de construire une configuration modulaire. Un nouveau fichier vide et tout tampon non enregistré dont le bloc settings vient d'un include sont touchés aussi.

La cause de fond : le CLI ne compile qu'un fichier racine, le LSP traite chaque tampon comme une racine. L'hypothèse est fausse.

H4 — L'analyseur de superposition de tableaux rate les deux fuites qui comptent reproduit

ArrayLayeringAnalyzer.cs:38 et :59

L'analyseur a été écrit pour couvrir le piège d'indexation des tableaux de .NET. Vérification faite contre un vrai ConfigurationBuilder, il laisse passer les deux cas les plus fréquents.

(a) Tableaux d'objets à longueur égale. La règle ne se déclenche que si l'overlay est plus court. Or :

base : "Svcs": [ { "Name": "a", "Port": 1 } ]
Dev  : "Svcs": [ { "Name": "b" } ]      → aucun avertissement

Longueurs identiques, donc silence — mais .NET fusionne l'index 0 champ par champ et Port = 1 fuit de la base vers Dev. L'heuristique par comptage d'éléments est structurellement aveugle à cette classe.

(b) Comparaison sensible à la casse. Cased et cased sont deux clés distinctes pour l'analyseur, une seule à l'exécution. Le point est systémique : CoverageAnalyzer et ValidateOverlayAgainstBase comparent également en Ordinal alors que la configuration .NET est OrdinalIgnoreCase. Ainsi Foo = 1 en base avec foo { Bar = 2 } en Dev passe sans erreur, alors que primitif-contre-objet est précisément le conflit que la nouvelle validation doit détecter — et l'avertissement de couverture émis est trompeur, annonçant une clé absente là où le vrai problème est une divergence de casse.

H5 — Débordement de pile non rattrapable dans le lexer corrigé

Lexer.cs:189 — return this.NextToken();

Hors crochets, ScanNewline se rappelait dans NextToken au lieu de boucler. Environ 6 000 lignes vides tuaient le processus :

Stack overflow.
   at Settex.Core.Lexer.Lexer.SkipWhitespace()
   at Settex.Core.Lexer.Lexer.NextToken()
   at Settex.Core.Lexer.Lexer.ScanNewline(Int32, Int32)
   ...   → exit 127, aucun diagnostic

Un StackOverflowException n'étant pas rattrapable en .NET, le catch global du compilateur était contourné. Le lexer tournant à chaque frappe, le serveur de langage mourait aussi. Le déclencheur est improbable à la main, trivial pour un .settex généré ou concaténé.

Correctif : ScanNewline renvoie désormais null quand le saut de ligne n'est pas significatif, et NextToken boucle au lieu de récurser. La pile ne croît plus avec le nombre de lignes vides : 200 000 lignes vides compilent, contre un plantage à 6 000. Deux tests verrouillent le comportement — l'absence de débordement, et le fait que les sauts de ligne restent bien émis à l'intérieur d'un tableau, où ils sont significatifs.

04Défauts moyens

Régressions de la campagne

M1 — La validation base/overlay rejette des overlays qui ne changent rien reproduit

Evaluator.cs:127 · :367

EvaluateNestedBlock matérialise un objet vide même quand toutes ses affectations sont écartées. settings { Foo = "bar" } avec env "Dev" { settings { Foo { X = 1 if false } } } échoue en Type mismatch … base is primitive, overlay is object, alors que l'overlay émis pour Foo est {}. Échec de compilation dur sur une configuration sémantiquement neutre.

M2 — null en base plus objet en overlay devient une erreur dure reproduit

AreCompatible classe null parmi les primitifs. Déclarer une clé à null en base pour la rendre visible, puis la remplir par environnement — la façon idiomatique de signaler une valeur obligatoire — est désormais refusé. null est pourtant le seul primitif qui n'engage aucun type. Décision de conception à trancher.

M3 — Les étendues d'affectation sont tronquées dès que la valeur est un opérateur reproduit

Parser.cs:535

BinaryOpNode porte la position de l'opérateur, MemberAccessNode celle de son premier segment. Pour B = 1 + 2, l'affectation s'arrête colonne 10 alors que le littéral 2 vit colonne 11-12. Survoler le 2 ne déclenche donc pas l'overlay — ce que le correctif d'étendue visait précisément à corriger. Casse aussi l'inclusion parent/enfant des plages LSP.

M4 — Un let de bloc env se retrouve hors de l'étendue de son parent reproduit

Parser.cs:405

Les let d'un env sont réinjectés dans la liste du bloc settings, qui conserve sa propre position : on obtient un enfant démarrant avant son parent. Tout consommateur supposant l'inclusion échoue ou produit une plage LSP invalide.

M5 — L'itérateur for corrompt l'évaluation des autres variables du scope

ScopeResolver.cs:222 · SettexHoverHandler.cs:320

L'itérateur est enregistré comme un LetNode synthétique dont la valeur est la collection. Le masquage et la navigation fonctionnent, mais BuildEvaluationScope lie alors le nom à la collection entière : toute autre variable du corps qui référence l'itérateur (let inner = svc.Name) est évaluée contre un tableau et affichée comme une valeur sûre et fausse.

Cohérence et robustesse

M6 — La CI contredit l'épinglage GA tout juste posé corrigé

.github/workflows/azure-static-web-apps.yml:39

Le workflow demandait explicitement dotnet-quality: 'preview', et son en-tête justifiait encore « it targets a preview .NET SDK ». Or global.json porte désormais allowPrerelease: false : un runner ne disposant que d'un SDK 10.0 preview ne satisfaisait plus le pin. Le filtre de chemins ayant exclu le commit global.json, rien n'avait cassé à la fusion : la rupture se serait déclenchée au prochain changement dans docs-site/.

Correctif : le workflow lit maintenant global-json-file, comme la CI. L'en-tête périmé est corrigé au passage — .NET 10 est GA ; la vraie raison du pré-build est qu'Oryx ne porte ni le SDK 10 ni le workload wasm-tools.

M7 — Refresh() est un lire-modifier-écrire qui annule une écriture concurrente reproduit

SettexDocument.cs:75

Deux écrivains coexistent désormais sur current : Update (frappe de l'utilisateur) et Refresh (déclenché par le changement d'un autre document). L'invariant de thread-safety documenté — un unique remplacement atomique — ne tient plus. Sous 300 mises à jour concurrentes, le tampon serveur recule silencieusement.

M8 — Un include introuvable n'est jamais retracé, l'erreur ne se lève donc jamais reproduit

Le suivi conserve ResolvedFiles, mais le seul fichier qui compte — celui qui n'a pas été trouvé — n'y figure pas par définition. Créer le fichier manquant ne rafraîchit pas le dépendant : le STX301 persiste jusqu'à ce que l'utilisateur modifie le fichier fautif.

M9 — Aucun gestionnaire didChangeWatchedFiles

L'extension enregistre bien un observateur de fichiers, mais aucun gestionnaire C# ne reçoit la notification. Un git checkout ou une modification hors éditeur d'un fichier inclus laisse les dépendants sur une analyse périmée indéfiniment. La résolution sur tampons ouverts ne couvre que le cas intra-éditeur.

M10 — Les avertissements de couverture n'ont aucune position et sont dupliqués reproduit

CoverageAnalyzer.cs:81

Les analyseurs travaillent sur le modèle évalué, qui a perdu les positions source : les diagnostics sont émis sans SourceLocation. En CLI la colonne Location est nue ; dans l'éditeur, les 11 avertissements de complete-v2-features.settex s'empilent tous sur le premier caractère. Et comme le modèle vient de l'AST aplati, les mêmes avertissements sont republiés sur chaque fichier incluant le fragment fautif. C'est la fonctionnalité phare du produit qui est la moins bien localisée.

M11 — Les déploiements de prévisualisation de PR échouent systématiquement

« This Static Web App already has the maximum number of staging environments ». Chaque PR affiche un rouge sans rapport avec le code — le bruit qui apprend à ignorer les alertes.

05Défauts mineurs

  • Reliquats de debug commités à la racine : debug_for.settex et trois appsettings*.json, tracés depuis d'anciens commits. Les appsettings contiennent du contenu fusionné, qui contredit visuellement le mode delta actuel. Rien dans .gitignore ne les couvre.
  • L'échappement $${ n'est documenté nulle part — zéro occurrence dans le README comme dans le site. Fonctionnalité livrée, invisible.
  • Changement de sens silencieux : il n'existe plus d'échappement produisant un $ littéral suivi d'une interpolation. "cost: $${price}" est passé de cost: 42 au littéral cost: ${price} sans avertir. Le contournement "$" + "${price}" fonctionne, mais mérite une note de version.
  • Ordre des environnements non déterministe dans les deux analyseurs : itération d'un Dictionary, y compris pour les listes citées dans le texte des messages.
  • Étendues absentes sur LetNode et IncludeNode — limitées au mot-clé, incohérent avec le traitement des affectations.
  • LocationToRange peut produire une plage inversée — endColumn n'est borné qu'à zéro, jamais à startColumn. Latent : aucune occurrence sur un AST réel aujourd'hui.
  • Décalage d'une colonne sur la fin exclusive dans IsPositionInLocation, et sur la position d'erreur de \$ dans le lexer.
  • Deux gestionnaires non protégés malgré l'annonce « les cinq » : les quatre surcharges de synchronisation de document et une surcharge de complétion.
  • cancellationToken ignoré partout : les cinq corps sont synchrones, donc la branche OperationCanceledException est du code mort — alors qu'un survol lance un évaluateur complet.
  • Sévérité Information étiquetée STX401 (le code d'erreur), le ternaire ne traitant que le cas Warning.
  • result?.dotnetPath accepté sans vérifier son existence dans l'extension : un chemin périmé part directement en commande, avec un ENOENT opaque au lieu du message actionnable prévu.
  • Survol incohérent sur l'en-tête d'un bloc : Server dans Server { … } ne renvoie rien, alors que Server.Port = … renvoie l'overlay complet.

06Ce qui a été vérifié solide

Une revue n'a de valeur que si elle dit aussi ce qui tient. Ces points ont été activement attaqués et n'ont rien donné :

Cohérence fusion / validation

Toutes les combinaisons objet / tableau / valeur / null ont été parcourues des deux côtés : AreCompatible est exactement la disjonction des branches acceptantes de MergeValues. Aucune divergence — la fusion non gardée du mode fusionné est réellement inatteignable par conflit.

Échappement $${

Quinze variantes sondées : escapes multiples, en fin de chaîne, collés à une interpolation réelle, imbriqués, non terminés. Aucun plantage, aucun décalage, aucune erreur mal positionnée. La garde est démontrée saine.

Décisions de couverture

Les deux règles arbitrées tiennent de bout en bout : une clé présente dans tous les environnements reste silencieuse, et l'ajout d'un nouvel environnement déclenche bien l'avertissement.

Étendues des blocs

settings, env, blocs imbriqués, for, objets tagués, tableaux vides, imbriqués et multi-lignes, valeur suivie d'une condition : aucune étendue inversée.

Includes cycliques

Pas de boucle de rafraîchissement infinie : A↔B et l'auto-inclusion terminent proprement sur STX301 Circular include detected. L'énumération concurrente est sûre.

Performance

Non-sujet à l'échelle réaliste : 30 dépendants sur un include partagé coûtent 0,8 ms par frappe ; le plus gros échantillon du dépôt, 0,5 ms.

08Conclusion

Le produit fait ce qu'il promet ; le processus qui devrait le garantir vient d'être posé.

Settex résout le problème qui l'a fait naître, et le résout bien : une source unique, un lint qui détecte la dérive entre environnements, une sortie en delta qui respecte le mécanisme appsettings de .NET plutôt que de le remplacer. Rien dans cette revue ne remet cela en cause.

Ce qu'elle établit, en revanche, c'est que la campagne de correctifs a atteint la limite de ce qu'une vérification manuelle peut tenir. Trois régressions élevées sont passées entre les mailles de 305 tests verts, dans le composant le moins couvert — le serveur de langage, où les défauts sont invisibles depuis une ligne de commande. La première priorité n'était donc aucun des défauts de cette liste : c'était le filet qui les aurait attrapés.

Ce filet est désormais posé, et il a justifié son existence avant même d'être fusionné : sa première exécution a mis au jour un défaut de portabilité que ni cette revue ni ses deux relecteurs n'avaient vu. C'est la meilleure démonstration possible de ce que trois mois de vérification manuelle ne peuvent pas remplacer.

Restent trois défauts élevés, tous du côté produit et tous concentrés là où le diagnostic est le plus coûteux : la confusion de positions entre fichiers dans le serveur de langage, l'erreur permanente sur les fragments d'include, et les angles morts de l'analyse de superposition. Ils peuvent maintenant être corrigés sans avoir à parier qu'aucun autre ne réapparaît en silence.
