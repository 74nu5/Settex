La quatrième revue s'est fermée à 9,4/10, tous constats traités. Cette cinquième en trouve cinq nouveaux défauts élevés, dont une perte de données silencieuse. La note baisse — et c'est la note précédente qui était trompeuse, pas celle-ci.

Ce que ce cinquième passage établit

La note mesurait la fermeture d'une liste, pas la densité de défauts. À chaque revue, les constats de la précédente ont été fermés puis la note remontée — mais chaque nouvelle relecture est allée plus profond que la précédente et a trouvé une nouvelle couche. Une liste vide ne signifie pas un logiciel sans défaut ; elle signifie que la dernière méthode d'inspection a épuisé ce qu'elle savait voir.

Deux des cinq défauts élevés viennent de correctifs de la campagne précédente : l'un parce qu'une correction se disait « systémique » sans l'être, l'autre parce que deux correctifs corrects séparément ouvrent un trou une fois combinés.

Mise à jour — depuis la publication de cette revue

Les cinq défauts élevés sont corrigés, en trois PR, chacune validée par la CI et accompagnée de tests dont l'échec a été vérifié sur le code non corrigé. 364 → 371 tests.

  • H1 · H2 — la comparaison de casse est terminée là où elle manquait : dictionnaire des environnements, PathExists, blocs imbriqués. Deux environnements ne différant que par la casse fusionnent désormais, comme le fait déjà le modèle de .NET.
  • H4 · H5 — les deux heuristiques sont remplacées par une seule règle tirée du comportement réel du provider : comparer les jeux de clés aplaties. Elle couvre les index de queue, les champs omis, les tableaux imbriqués et l'élément objet remplacé par un primitif.
  • H3 — un diagnostic n'est plus jamais écarté. Le compromis est assumé : la duplication revient, et c'est préférable à une disparition.

La promesse centrale tient toujours : une clé posée dans un environnement et oubliée ailleurs produit un avertissement nommant la clé, les environnements et l'action, avec trois fichiers en delta. Et le filtrage de diagnostics introduit dans le serveur de langage n'a pas fui dans le compilateur — vérifié explicitement, c'était un risque réel.

03Défauts élevés

H1 — Deux environnements ne différant que par la casse détruisent un overlay corrigé

Evaluator.cs:63 · JsonWriter.cs:71

Le dictionnaire des overlays compare en ordinal : env "Dev" et env "dev" sont deux environnements. Sur Windows, ils s'écrivent dans le même fichier.

✓ Successfully generated 3 file(s):
  → appsettings.json
  → appsettings.Dev.json
  → appsettings.dev.json     ← le même fichier

fichiers réellement présents : 2
appsettings.Dev.json : { "A": 200 }   ← l'overlay Dev (A = 100) a disparu

Perte de données silencieuse, annoncée comme un succès, avec une liste de fichiers qui ment. Le comportement diverge aussi entre plateformes : Linux écrit bien deux fichiers.

H2 — := est sensible à la casse et écrase la valeur de base corrigé

Evaluator.cs:309 · PathExists à :351

PathExists compare en ordinal, donc ne voit pas une clé de base orthographiée différemment :

base     : Timeout = 30      Db { Host = "base" }
env Prod : timeout := 99     db { Host := "prod" }

overlay produit : { "timeout": 99, "db": { "Host": "prod" } }

L'opérateur dont le contrat entier est « ne pas écraser » écrase.

H1 et H2 ont la même origine : un correctif qui se disait systémique

La campagne précédente a rendu insensibles à la casse les trois analyseurs et ValidateOverlayAgainstBase, en qualifiant le point de « systémique ». L'assistant TryGetPropertyIgnoreCase n'a été câblé que dans la validation — ni dans PathExists, ni dans la recherche de base des blocs imbriqués — et le dictionnaire des environnements n'a jamais été touché. Le diagnostic était juste ; le correctif s'est arrêté là où le relecteur avait pointé, au lieu d'aller là où le diagnostic menait.

H3 — Un avertissement de dérive disparaît quand la clé vit dans un fragment sans settings corrigé

SettexDocument.cs:192 et :204

Deux correctifs de la campagne, corrects séparément, ouvrent un trou une fois combinés :

  • Dans la racine, un diagnostic dont la clé n'est pas assignée dans ce fichier est écarté — « il est signalé là-bas, sur l'affectation elle-même ».
  • Dans le fragment, il ne l'est pas : sans bloc settings, les analyseurs ne tournent pas du tout.

Découper un fichier en include supprime donc, dans l'éditeur, l'avertissement même que Settex existe pour produire. Nuance importante : le compilateur, lui, avertit toujours — la divergence est entre l'éditeur et la ligne de commande.

H4 — L'analyse de superposition rate les tableaux imbriqués dans un élément objet corrigé

ArrayLayeringAnalyzer.cs:158

CollectLeaves traite un tableau comme une feuille : si l'override redéfinit le champ, il est réputé couvert. Mais .NET superpose aussi ce tableau interne, par index.

base : Svcs = [ svc { Name="a"  Tags=["t1","t2","t3"] } ]
Prod : Svcs = [ svc { Name="a2" Tags=["x"] } ]

aucun avertissement · runtime : Svcs:0:Tags = ["x","t2","t3"]

C'est exactement la fuite que l'analyseur existe pour attraper, un niveau plus bas.

H5 — Un élément objet écrasé par un primitif est réputé « remplacé » corrigé

ArrayLayeringAnalyzer.cs:96

Le continue saute l'index dès que l'un des deux éléments n'est pas un objet, avec un commentaire affirmant que le primitif écrase franchement. Faux : base [ p { Name="base" Port=1 } ] surchargée par [ 5 ] donne à l'exécution Prim:0 = 5 et Prim:0:Name = base. La garde ne devrait sauter que si l'élément de base n'est pas un objet.

04Défauts moyens

M1 — Deux blocs imbriqués de même nom s'écrasent au lieu de fusionner reproduit

settings { Db { Host = "h" } Db { Port = 5 } } émet { "Db": { "Port": 5 } } — Host est perdu sans diagnostic, alors que le modèle documenté pour les blocs settings multiples est la fusion profonde. Pré-existant, mais sur la ligne même que la campagne a réécrite.

M2 — Le saut des blocs vides repose sur une justification fausse reproduit

Le commentaire affirme qu'un objet vide ne produit aucune clé à l'exécution. Le relecteur a nourri { "A": {}, "B": [] } à un vrai ConfigurationBuilder : A et B ressortent comme des clés réelles. La décision reste bonne pour le conflit qu'elle corrigeait, mais elle supprime aussi les blocs explicitement vides, ce que le raisonnement ne couvrait pas.

M3 — « Rechercher les références » ignore tout ce qui est dans une boucle reproduit

ForNode implémente IArrayElement mais pas IExpression, et le collecteur filtre sur IExpression : ni le corps ni la collection ne sont parcourus. Une variable utilisée uniquement dans une boucle renvoie zéro référence. La résolution de portée, elle, gère correctement les boucles — c'est le collecteur seul.

M4 — Le survol d'un segment de chemin répété résout le mauvais objet reproduit

IndexOf(word) prend la première occurrence. Sur settings { A { B { A = 1 } } }, survoler le A le plus interne affiche l'overlay de A au lieu de la valeur de A.B.A. L'ajout du survol d'en-tête de bloc, qui produit des chemins multi-segments, élargit le cas.

M5 — Un fichier racine sans settings est propre dans l'éditeur et échoue au build

La relaxation nécessaire aux fragments masque aussi l'erreur pour un fichier destiné à être une racine. Rien ne distingue les deux intentions, donc il n'y a pas de correctif gratuit — mais la divergence éditeur/compilateur est réelle et n'est documentée nulle part.

M6 — Sorties générées commitées et périmées dans le projet d'exemple reproduit

tests/IntegrationTests/SampleProject/ contient six appsettings*.json commités dont le contenu est entièrement fusionné, soit ce que le compilateur émettait avant le mode delta. Recompilation à l'appui, ils diffèrent de ce que l'outil produit aujourd'hui. Le répertoire n'est de plus pas dans la solution, donc ni construit ni vérifié par la CI. C'est la classe de défaut nettoyée à la racine du dépôt — la campagne n'avait regardé que la racine.

M7 — Le workflow de déploiement n'a pas de bloc permissions et casse sur les PR de fork

C'est pourtant lui qui détient un secret de déploiement, alors que ci.yml restreint correctement à contents: read. Il se déclenche par ailleurs sur pull_request et lit un secret vide pour un fork, ce qui fait échouer l'étape de déploiement.

05Défauts mineurs

  • Fins de ligne CR seules : tous les diagnostics annoncent la ligne 1. ScanNewline gère le \r nu, mais Advance() n'incrémente la ligne que sur \n.
  • Les erreurs d'include introuvable ou illisible n'ont pas de position — ancrées en 0:0, alors que IncludeNode vient justement de recevoir une vraie étendue.
  • Des tableaux frères ne différant que par la casse s'effondrent dans l'analyseur : le dernier écrase le premier dans la table, ce qui masque un raccourcissement réel.
  • TryGetPropertyIgnoreCase dépend de l'ordre d'insertion quand deux clés frères ne diffèrent que par la casse.
  • La complétion n'offre que les let de premier niveau — ni variables d'environnement, ni itérateur de boucle, sans aucune conscience de la position.
  • didClose peut laisser des soulignements périmés : l'effacement des diagnostics vient après une boucle dont une exception serait avalée par la garde.
  • Deux URI ouverts pour le même fichier rendent la résolution d'includes non déterministe (dépendante de l'ordre d'énumération).
  • cancel-in-progress annule la CI d'un commit déjà fusionné quand la fusion suivante arrive.
  • client.start() est une promesse flottante dans l'extension, et ni le client ni l'observateur ne sont ajoutés à context.subscriptions.

06Ce qui a été attaqué et a tenu

Concurrence du serveur

Aucun ordre de verrous à inverser : Refresh tient son verrou et ne fait qu'une lecture volatile sans verrou sur les autres documents. 300 DidChange concurrents sur deux fichiers s'incluant mutuellement : 18 ms, textes finaux corrects, aucun recul.

Le débordement de pile

Réellement corrigé : 50 000 lignes vides compilent. La boucle ne peut pas tourner à vide — ScanNewline n'est atteint que sur un vrai saut de ligne, qui est toujours consommé.

Étendues du parser

this.Previous ne peut jamais précéder le chemin. Aucune étendue inversée ni de largeur nulle trouvée, et aucun consommateur ne supposait que le bloc démarre sur settings.

Asymétrie du null

Implémentée correctement et de façon cohérente : base nulle plus overlay objet compile, overlay nul sur base structurée échoue — identiquement en mode delta et fusionné.

Tampons non enregistrés

Pas de trou : sans chemin, les includes ne sont pas résolus du tout, donc le défaut par défaut de IsFromSameFile n'a rien à confondre.

Le résolveur d'includes

Enregistrer le chemin avant le chargement ne casse ni la déduplication en losange, ni la détection de cycle, ni le double include d'un fichier manquant.

08Conclusion

Le produit tient sa promesse ; c'est la confiance dans la note qui doit baisser, pas la confiance dans l'outil.

Rien ici ne remet en cause ce que Settex fait : le scénario fondateur fonctionne, la sortie delta respecte le mécanisme appsettings de .NET, la CI est saine et couvre deux plateformes, et la concurrence du serveur de langage a résisté à une attaque directe.

Ce que cette cinquième relecture établit tient en une phrase : une liste de constats vide ne mesure pas l'absence de défauts, elle mesure l'épuisement d'une méthode. Quatre revues ont chacune fermé la liste de la précédente, et chacune a été suivie d'une nouvelle qui trouvait plus profond. La note montait parce qu'elle suivait la fermeture ; elle redescend ici parce qu'un regard neuf a trouvé une couche que les précédents n'atteignaient pas.

Deux des cinq défauts élevés viennent directement de la campagne précédente, pour deux raisons qui méritent d'être distinguées. L'un parce qu'un correctif s'est arrêté aux trois endroits que le relecteur avait nommés, alors que son propre diagnostic disait « systémique » — corriger là où l'on a été pointé, pas là où le raisonnement mène. L'autre parce que deux correctifs justes séparément se combinent en un trou : aucune relecture de l'un ou de l'autre ne pouvait le voir, seule une relecture de l'ensemble le pouvait.

Les constats marqués « reproduit » l'ont été sur du code exécuté ; les effets d'exécution ont été vérifiés contre un vrai ConfigurationBuilder et contre des harnais liés aux types réels du serveur.
