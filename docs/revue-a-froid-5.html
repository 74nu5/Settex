Settex — changer de prise

Cinq revues ont lu le code. Celle-ci ne le lit pas : elle compare la sortie de Settex à ce qu'un vrai ConfigurationBuilder en fait, et elle mute le code de production pour voir si les 381 tests s'en aperçoivent.

La cinquième revue concluait qu'une liste vide mesure l'épuisement d'une méthode, pas l'absence de défauts. Cette sixième le vérifie en changeant de méthode — et trouve immédiatement un défaut qu'aucune des cinq précédentes ne pouvait voir : Settex peut produire, sans le moindre avertissement, un appsettings.json que .NET refuse de charger.

Pourquoi cette revue n'a pas relu le code

Les revues 4 et 5 ont trouvé chacune exactement cinq défauts élevés, la quatrième s'étant pourtant fermée « liste vide ». Le taux de découverte n'a pas décliné. Refaire une sixième lecture adverse aurait probablement produit cinq constats de plus, sans rien apprendre de neuf sur la méthode.

Deux prises différentes ont donc été employées : un test différentiel — compiler des dizaines de configurations et comparer ce que le mode fusionné annonce, ce qu'un vrai ConfigurationBuilder produit, et ce que Settex a averti — et un test par mutation du code de production, pour mesurer ce que les 381 tests protègent réellement. Les deux ont trouvé des choses structurellement invisibles à la lecture.

02Le défaut critique

C1 — Deux clés frères ne différant que par la casse produisent un fichier illisible par .NET critique

reproduit · vérifié contre un vrai ConfigurationBuilder

settings {
    Foo = "one"
    foo = "two"
}

Settex : ✓ Successfully compiled   0 avertissement
sortie : {"Foo":"one","foo":"two"}

.NET   : System.FormatException: A duplicate key 'foo' was found.
         → l'application ne démarre pas

Le compilateur annonce un succès et émet un fichier que le système de configuration refuse en bloc. Ce n'est pas une valeur fausse : c'est l'absence totale de configuration, au démarrage, en production.

Settex est incohérent avec lui-même sur ce point. := résout la casse sans y être sensible, le contrôle de dérive est insensible à la casse, la validation de conflit de type l'est aussi — trois campagnes de correctifs y ont veillé. Mais l'affectation simple = construit un magasin de clés sensible à la casse, si bien que seul l'écrivain JSON voit deux clés.

Le déclencheur réaliste n'est pas une faute de frappe : ce sont deux includes. Un fichier définit List, un autre list, un troisième inclut les deux. Settex émet deux avertissements sans rapport et aucun indice du défaut fatal.

03Défauts élevés

H1 — Le mode fusionné produit lui aussi un fichier illisible reproduit

Base Server.Port et env server.port : en mode delta la superposition fonctionne (.NET est insensible à la casse). En mode --merged, les deux orthographes sont écrites dans le même fichier, qui lève A duplicate key 'server:port'. La configuration effective annoncée par ce mode n'est donc pas seulement fausse — elle n'existe pas.

H2 — Une clé contenant un point ne peut pas être exprimée, et l'exemple ASP.NET Core du README produit une configuration inopérante reproduit

Logging { LogLevel { Microsoft.AspNetCore = "Warning" } }

produit : {"Logging":{"LogLevel":{"Microsoft":{"AspNetCore":"Warning"}}}}
attendu : {"Logging":{"LogLevel":{"Microsoft.AspNetCore":"Warning"}}}

Le point est toujours un séparateur de chemin, jamais un caractère. Lié via LoggerFilterOptions, la règle produite porte la catégorie Microsoft:AspNetCore, qui ne correspond à rien : le filtre de journalisation demandé est silencieusement ignoré.

Il n'existe aucun contournement : les clés entre guillemets sont refusées par le parser, dans les deux positions. Logging:LogLevel:<Catégorie> étant le réglage ASP.NET Core le plus courant, Settex ne peut pas exprimer l'appsettings.json que dotnet new webapi génère — et échoue en silence plutôt que bruyamment. L'exemple du README et samples/SampleWebApi embarquent tous deux le défaut.

H3 — Deux environnements ne différant que par la casse fusionnent sans avertir régression de correctif

La campagne précédente a corrigé une perte de données ici — les deux écrivaient dans le même fichier et l'un écrasait l'autre — en les faisant fusionner, au motif que le modèle de .NET les considère comme un seul environnement. Le raisonnement tient pour IsDevelopment(), mais pas pour le nom de fichier : sur Linux, ASPNETCORE_ENVIRONMENT=dev ne charge aucun fichier, puisque seul appsettings.Dev.json existe. Les deux overlays disparaissent.

Le correctif a donc échangé un mode de défaillance contre un autre, sans avertissement dans aucun des deux cas. Ce qui manque n'est pas la fusion — c'est le diagnostic.

H4 — Trois invariants de valeur ne sont protégés par aucun test mutations survivantes

Mutées une par une, ces trois lignes ne font échouer aucun des 381 tests :

  • InvariantCulture → CurrentCulture dans l'interpolation et la concaténation. Sur cette machine fr-FR, la sortie devient "rate=1,5". Aucun test n'interpole un nombre fractionnaire — vérifié indépendamment.
  • L'encodeur JSON relâché → Default : &, < et tout caractère non-ASCII deviennent &… dans tous les fichiers générés.
  • L'égalité de chaînes → insensible à la casse : change quelles affectations conditionnelles se déclenchent, dans chaque configuration de chaque utilisateur.

H5 — Deux tests de régression ne peuvent pas échouer, dont un écrit pendant cette campagne reproduit

Hover_OnTheColumnJustPastAnAssignment_DoesNotReturnTheOverlay passe dans les deux sens : une garde de nullité protège la seule assertion, et la source choisie fait que n'importe quel overlay contient le mot cherché — y compris exactement le débordement d'une colonne que le test existe pour interdire.

DetectCycle_WithoutCycle_ReturnsFalse est une tautologie : sur un résolveur neuf la pile est vide, la méthode renvoie false quel que soit son argument.

Le motif est constant, et il s'est manifesté trois fois dans cette campagne : une garde de nullité ou une sortie anticipée transforme un test de régression en test toujours vert. Quatre autres tests portent un nom qui promet plus que leur corps ne vérifie.

H6 — L'exemple phare de la page d'accueil est faux, et l'exemple d'application ne compile pas reproduit

Le bloc étiqueté « appsettings.Production.json (généré) » montre une clé que la sortie delta ne contient pas — contredisant, sur la même page, la fonctionnalité mise en avant. Et samples/SampleWebApi, vers lequel le README pointe, échoue à la compilation tout en illustrant l'anti-patron que le README déconseille.

04Défauts moyens

  • --no-coverage-check désactive aussi l'avertissement de superposition de tableaux, présenté comme inconditionnel par le README. Deux dangers distincts derrière un seul drapeau — un défaut de conception, pas seulement de documentation.
  • ~1 320 lignes de gestionnaires du serveur de langage n'ont aucun test, dont SettexCompletionHandler (699 lignes), la fonctionnalité la plus visible des deux extensions.
  • Sept mutations de plus survivent : portée de l'itérateur for — dont le test qui porte ce nom ne vérifie pas la chose —, détection des let dupliqués, garde de division par zéro (qui survit à cause d'une assertion non typée), court-circuit de and/ou, normalisation des fins de ligne en sortie.
  • Sept Assert.ThrowsAsync non typés acceptent n'importe quelle exception, y compris un plantage. Le style typé est utilisé ailleurs dans les mêmes fichiers.
  • Dérive documentaire : emplacement de sortie par défaut, « les virgules sont optionnelles » (vrai seulement à travers un saut de ligne), sorties d'exemples périmées, spécification annonçant « un seul bloc settings » alors que plusieurs fusionnent.

05Ce qui a tenu

Il faut le dire aussi nettement que le reste, car c'est un résultat solide et il est le produit direct des campagnes précédentes.

Détection des fuites de tableaux

Aucun faux négatif trouvé, sur une quinzaine de formes : overrides plus courts, égaux, plus longs, tableaux d'objets, tableaux imbriqués dans un élément, tableaux de tableaux, tableaux issus d'une boucle, apportés par un include, ou dont la clé diffère par la casse. Chaque désaccord entre mode fusionné et superposition réelle était averti.

Les invariants centraux, sous mutation

34 mutations sur 44 tuées, dont toutes celles qu'on attendrait en trou : dérive, fuites de superposition, delta contre fusionné, cycles et losanges d'includes, validation de conflit de type, insensibilité à la casse des analyseurs et de :=. Plusieurs tuées par 3 à 10 tests.

Valeurs et types

null en base, en overlay, dans un élément de tableau ; échappements, unicode et paires de substitution ; long.MaxValue, flottants, division ; entiers hors plage et échappements invalides en erreur dure. Delta et superposition réelle concordent partout.

Documentation vérifiée

Une part substantielle tient exactement, compilateur à l'appui : tous les opérateurs et leur précédence, la section d'échappement entière y compris le piège $${, l'exemple d'include au caractère près, la détection de cycles, la sémantique de let, :=, if et for, le texte de l'avertissement de dérive, et tous les drapeaux du CLI.

07Conclusion

Changer de méthode a trouvé en une passe ce que cinq lectures n'avaient pas atteint.

Le résultat le plus utile de cette revue n'est pas la liste — c'est la démonstration de sa thèse. Cinq revues successives ont lu le code de plus en plus finement, en fermant chaque fois la liste précédente. Aucune n'a trouvé qu'un fichier généré pouvait être illisible, parce qu'aucune ne le donnait à lire à .NET. Aucune n'a trouvé que trois invariants de valeur n'étaient protégés par rien, parce qu'aucune ne mutait le code pour interroger les tests.

La contrepartie mérite d'être dite avec la même netteté : là où les campagnes précédentes ont travaillé, le travail tient. La détection des fuites de tableaux n'a pas produit un seul faux négatif sur quinze formes, et les invariants centraux ont survécu à 34 mutations sur 44. Ce n'est pas un logiciel fragile ; c'est un logiciel dont les zones vérifiées sont solides et dont les zones jamais interrogées ne l'avaient jamais été.

Enfin, un constat qui vaut pour la suite : H3 est une régression d'un correctif de la campagne précédente, et H5 concerne un test écrit pendant cette même campagne. Corriger sous pression produit des correctifs qui échangent un mode de défaillance contre un autre, et des tests qui rassurent sans vérifier. La réponse n'est pas de corriger plus vite.
