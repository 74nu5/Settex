01Méthode & périmètre

L'objectif était une revue indépendante : ne pas se fier aux commentaires, aux noms de tests, ni aux rapports antérieurs — et reproduire les défauts avant de les affirmer.

  • Vérifié dynamiquement : build de la solution (0 avertissement), 236 tests (215 core + 15 LSP + 4 MSBuild + 2 extension) tous au vert, et des dizaines de compilations CLI de sondage (messages d'erreur, cas limites, bugs).
  • Revue de code du pipeline complet (Lexer → Parser → IncludeResolver → Evaluator → Merger → JsonWriter), du serveur LSP, des deux extensions et du site de docs.
  • Reproduction : chaque bug listé au §02 a été déclenché au CLI ou constaté dans le paquet produit, pas seulement lu.

Le projet compte ~9 200 lignes (C#/Razor/TS) : compilateur (~3 510), serveur LSP (~2 985), extensions (~1 330), site de docs Blazor (~1 095), tests (~6 150).

02Bugs reproduits

Défauts confirmés par reproduction — non couverts par les tests existants.

✅ Corrigé — L'extension VS Code ne s'activait pas une fois installée
.vscodeignore excluait node_modules/**, mais out/extension.js exécute require("vscode-languageclient/node") (dépendance runtime). Le VSIX produit contenait 0 entrée node_modules → Cannot find module à l'activation. Corrigé : on cesse d'ignorer node_modules (vsce embarque les deps de prod et élague les dev-deps), on exclut les fichiers de dev, et on corrige l'URL placeholder du dépôt. Le VSIX repackagé embarque désormais vscode-languageclient/node (vérifié). Reste conseillé : bundler via esbuild (perf).
extensions/vscode-settex/.vscodeignore · package.json
✅ Corrigé — La soustraction sans espaces
Le lexer rendait tout - suivi d'un chiffre glouton en littéral négatif. 5-3 → erreur. Corrigé : le lexer émet toujours un token Minus et laisse le parser gérer le moins unaire (ParseUnary) et binaire (ParseTerm). Vérifié : 5-3→2, -5→-5, a-1→9, 10-3-2→5, (5-3)*2→4. Tests de non-régression ajoutés (lexer + end-to-end).
src/Settex.Core/Lexer/Lexer.cs (ScanMinus)
✅ Corrigé — Les let d'un bloc imbriqué fuyaient dans la portée parente
Un let défini dans un bloc imbriqué était visible en dehors et écrasait un homonyme global (Box { let host = "inner" } polluait le global). Corrigé : EvaluateNestedBlock crée désormais un scope enfant (CreateChild()) — les let imbriqués restent locaux mais lisent toujours les variables parent. Vérifié : plus de fuite ni d'écrasement, lecture du parent OK ; 3 tests de non-régression ajoutés.
src/Settex.Core/Evaluation/Evaluator.cs (EvaluateNestedBlock)
✅ Corrigé — Interpolation : jetons en trop ignorés, diagnostics faux
ParseInterpolatedString gardait uniquement a dans "${a b}" et ignorait b silencieusement ; le sous-lexer était créé sans filePath → toute erreur dans un ${…} pointait une localisation fantôme (ligne 1/col 1). Corrigé : après ParseExpression(), tout jeton résiduel lève une erreur ("${a b}" → « unexpected 'b' ») ; le filePath est propagé et toute erreur d'interpolation est ré-ancrée sur la localisation réelle de la chaîne. Vérifié : "${a b}" échoue à la bonne ligne, "${port + 100}" fonctionne, erreur interne ré-ancrée ; 2 tests de non-régression ajoutés.
src/Settex.Core/Parser/Parser.cs (ParseInterpolatedString)
✅ Corrigé — Includes en losange non dédupliqués · robustesse récursion
Le champ visited était du code mort : un include en losange (A→B, A→C, B→D, C→D) appliquait D deux fois ; la comparaison de chemins était sensible à la casse (sur Windows Inc.settex ≠ inc.settex → dédup/cycle raté) ; aucune borne de profondeur → StackOverflowException non rattrapable. Corrigé : l'ensemble included est désormais consulté (dédup include-once), la comparaison suit la casse du système de fichiers (PathComparer — insensible sur Windows), et une borne MaxIncludeDepth (64) lève une IncludeException localisée au lieu de déborder la pile. Vérifié : losange → D appliqué une fois (JSON & nombre de statements), chaîne de 70 → erreur « depth limit », dédup insensible à la casse sur Windows ; 3 tests de non-régression ajoutés.
src/Settex.Core/Resolution/IncludeResolver.cs (included, PathComparer, MaxIncludeDepth)
✅ Corrigé — Overflow numérique → « Unexpected error » sans localisation
Un littéral flottant hors plage decimal (ou une arithmétique qui débordait) levait une OverflowException non typée tombant dans le catch-all du compilateur → message générique sans ligne/colonne. Corrigé : un helper CheckedNumber enveloppe le cast littéral et les quatre opérations (+ − × ÷) et convertit l'OverflowException en EvaluatorException localisée. Vérifié : littéral hors plage → erreur à (2,9), big * 1000.0 → erreur à l'opération, décimal large mais valide toujours accepté ; 3 tests de non-régression ajoutés.
src/Settex.Core/Evaluation/ExpressionEvaluator.cs (CheckedNumber)

Compilateur

Points forts réels : pipeline en phases isolées, exceptions typées par phase, records immuables, deep-merge multi-blocs correct et bien testé, écriture conditionnelle idempotente, et surtout des diagnostics de qualité (voir §04). Faiblesses : les bugs du §02, plus des choix ouverts — les fichiers appsettings.{Env}.json contiennent la config complète mergée (base dupliquée dans chaque fichier, redondant avec le layering natif de .NET), les let sont hoistés (utilisables avant déclaration), et un peu de code mort subsiste (Parser.ParseValue, Evaluator.EvaluateLiteral).

Serveur de langage — la plus grande sur-évaluation du 1er rapport

Correctif du 1er rapport à nuancer
Le 1er rapport annonçait SettexDocument « rendu thread-safe ». C'est vrai pour l'infrastructure (snapshot immuable + volatile), mais le snapshot n'est capturé une fois que dans 1 handler sur 4 (la complétion). Hover, Definition et References lisent Text puis Ast en accès séparés → un Update concurrent peut apparier un texte N à un AST N+1. Le commentaire du fichier affirme le contraire.
  • Heuristiques de position non fiables : LocationToRange est mono-ligne (SourceLocation n'a pas d'EndLine/EndColumn) → ranges de symboles/diagnostics faux dès qu'un nœud est multi-ligne. ContainsPosition avale ~1000 lignes en dur après le début d'un scope → résolution de portée incorrecte avec plusieurs env.
  • Navigation cross-fichier cassée : les includes sont inlinés, mais Definition renvoie l'URI du fichier courant avec les lignes du fichier inclus → pointe au mauvais endroit.
  • Performance : reparse complet + File.ReadAllText de tous les includes à chaque frappe ; évaluation complète du modèle deux fois par complétion, une par hover, sans cache ; et surtout des arguments de log Trace évalués systématiquement (string.Join, ToJsonString) même quand le trace est éteint — coût réel en production, à protéger par IsEnabled.
  • URI → chemin maison (Substring(8)) : ne décode pas le percent-encoding, casse la racine / sur Linux, ignore l'UNC. DocumentUri.GetFileSystemPath() d'OmniSharp existe et n'est pas utilisé.
  • Manques : rename, code actions, formatting, semantic tokens, folding, sync incrémental.

Extensions

Visual Studio (net472) : l'ingénierie difficile est de bonne qualité — le compile-on-save gère correctement la concurrence (disposal déterministe du CTS, remove atomique clé/valeur), le kill de process, un timeout de 60 s, la lecture concurrente des flux et le marshalling UI. ✅ Les défauts du client LSP sont corrigés : stderr est désormais drainé en continu (BeginErrorReadLine → Debug, plus de deadlock), le process serveur est stocké et tué via Dispose (plus d'orphelin à la fermeture de VS), et le démarrage est protégé par un try/catch (dégradation propre si dotnet absent).

VS Code : au-delà du VSIX cassé (§02), métadonnées placeholder (github.com/yourusername/settex), versionnage désaligné avec l'extension VS (0.1.0 vs 1.2.0), et fichiers parasites embarqués (test.settex, TESTING.md).

Commun aux deux : dépendance dure au runtime .NET 10 (preview) sur la machine de l'utilisateur, sans détection ni message actionnable si absent — le principal frein d'adoption. Aucune CI de build/test/package pour les extensions.

Site de documentation

Proprement codé (service de localisation, composant de base, isolation build volontaire, config Azure SWA correcte, CI de déploiement). Mais : i18n par duplication (chaque page contient FR et EN en dur — dérive garantie à terme, pas de .resx), FOUC de langue (l'anglais s'affiche avant la bascule FR au chargement), <html lang="en"> figé, aucun SEO (WASM 100 % client, contenu documentaire invisible aux crawlers), et la classe lang-* de CodeBlock suggère une coloration syntaxique qui n'existe pas (aucun moteur branché).

04Évaluation fonctionnelle

Du point de vue de l'utilisateur : qu'est-ce qui marche vraiment, et est-ce agréable ?

Ce qui fonctionne bien

  • Le flux principal est solide. Un .settex réaliste (variables, expressions, interpolation, for, overlays, includes modulaires) compile de bout en bout et produit un JSON correct.
  • Les diagnostics sont une vraie force (vérifié sur ~15 cas d'erreur) : localisation ligne/colonne précise et messages lisibles — variable non définie, condition non booléenne, division par zéro, include circulaire, accès membre sur non-objet, chaîne non terminée (« Unterminated string literal »), entier hors plage (« Invalid integer literal '…' »), token inconnu. Peu de langages d'auteur soignent autant ce point.
  • La proposition de valeur est claire : DRY par overlays + deep-merge, syntaxe lisible, intégration MSBuild transparente.

Pertinence produit

Settex concurrence des approches établies (transformations de config .NET, variables d'environnement + dotnet user-secrets, ou des langages comme CUE / Dhall / Jsonnet). Sa niche — un DSL lisible avec overlays et deep-merge pour .NET — est réelle mais étroite. Le choix de générer des fichiers d'env complets (plutôt que des deltas) le met en tension avec le modèle de superposition natif de .NET, ce qui mérite d'être tranché. L'adoption dépendra d'une expérience « ça marche du premier coup » irréprochable — que le bug de soustraction et le VSIX cassé compromettent aujourd'hui.

05Évaluation critique

Settex reste un projet d'auteur au-dessus de la moyenne sur l'ingénierie : découpage en phases textbook, code discipliné (warnings-as-errors), un serveur LSP maison avec un hover multi-environnement original, et une exigence rare sur les messages d'erreur. La consolidation récente est réelle : includes modulaires, expressions complètes, extensions buildables, doc réalignée et vérifiée.

Mais une lecture à froid ramène à la réalité d'un alpha : le périmètre communiqué comme fini cache des défauts de correction que le style de tests dissimule. Le motif était récurrent — l'infrastructure correcte existe (snapshot thread-safe, scoping par VariableScope, champ anti-losange) mais n'était pas branchée au moment de la revue : le block-scoping n'appelait jamais CreateChild et le champ anti-losange n'était jamais lu (désormais corrigés ✅) ; côté LSP le snapshot n'est toujours utilisé qu'à un endroit. Le code ressemblait à ce qu'il devrait faire plus qu'il ne le faisait — d'où l'importance de ne pas croire les commentaires.

Ce qui impressionne

  • Diagnostics précis & lisibles
  • Deep-merge correct & testé
  • Compile-on-save VS robuste
  • Discipline d'architecture

Ce qui inquiète

  • Bugs de langage non testés ✅
  • Infra correcte mais non branchée
  • LSP heuristique & concurrent
  • Tests à assertions faibles

Ce qui bloque

  • VSIX VS Code non fonctionnel ✅
  • stderr LSP non drainé (VS) ✅
  • Dépendance .NET 10 preview
  • Pas de publication marketplace

Conclusion
Le socle mérite d'être poursuivi : peu de projets d'auteur atteignent ce niveau de soin architectural et de diagnostics. Mais pour prétendre à « prêt à l'usage », il faut fermer l'écart entre ce que le code annonce et ce qu'il fait — réparer le VSIX VS Code, éliminer les bugs de langage non testés, brancher l'infrastructure déjà écrite (snapshot, scoping, anti-losange), et durcir les tests là où ils ferment les yeux. La priorité n'est ni les fonctionnalités ni le polish : c'est la correction.
